## Task 20: Troubleshoot NetworkPolicy

```bash
kubectl config use-context cluster1-admin@cluster1

# Investigate the situation
kubectl get networkpolicy allow-client -n trouble-final -o yaml
# The policy allows ingress to server-pod only from pods with label role=correct-client

kubectl get pod client-pod -n trouble-final --show-labels
# client-pod has label role=wrong-client — this doesn't match the policy

# Fix: update the client-pod label to role=correct-client
kubectl label pod client-pod -n trouble-final role=correct-client --overwrite

# Verify the label is updated
kubectl get pod client-pod -n trouble-final --show-labels

# Verify the connection now works
kubectl exec client-pod -n trouble-final -- curl http://server-svc -s --connect-timeout 3
```

**Verify the NetworkPolicy is still in place:**
```bash
kubectl get networkpolicy allow-client -n trouble-final
```

**Key concepts:**
- NetworkPolicies use label selectors — if pod labels don't match, traffic is blocked
- `kubectl label pod <name> <key>=<value> --overwrite` updates an existing label
- The NetworkPolicy must remain — the fix is the pod label, not removing the policy
- After fixing the label, the client pod should be able to reach server-svc and get HTTP 200
